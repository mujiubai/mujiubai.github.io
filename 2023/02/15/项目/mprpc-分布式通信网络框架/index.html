1.2 单机服务器、集群和分布式

单机服务器

缺点：

  • 受限于硬件资源，聊天服务器所能承受的用户并发量有限
  • 任意模块的修改，都会整个项目代码重新编译部署
  • 系统中，有的任务是CPU密集型，有的则是IO密集型，造成各模块对硬件资源的需求不一样

集群

每一台服务器独立运行一个工程的所有模块。

  • 优点：并发量提升
  • 缺点：代码还需重新编译且需要多次部署，有的模块如后台管理不需要高并发不需要多次部署

分布式

一个工程拆分了很多模块，每一个模块独立部署运行在一个服务器主机上，所有服务器协同工

作共同提供服务，每一台服务器称作分布式的一个节点，根据节点的并发要求，对一个节点可以再做节

点模块集群部署。

1.3 RPC通信原理

  • 黄色部分：设计rpc方法参数的打包和解析，也就是数据的序列化和反序列化，使用Protobuf。

  • 绿色部分：网络部分，包括寻找rpc服务主机，发起rpc调用请求和响应rpc调用结果，使用muduo网络库和zookeeper服务配置中心（专门做服务发现）。

mprpc框架主要包含以上两个部分的内容

1.4 Protobuf

protobuf (protocol buffer) 是谷歌内部的混合语言数据标准。通过将结构化的数据进行序列化(串行化)，用于通讯协议、数据存储等领域的语言无关、平台无关、可扩展的序列化结构数据格式。

1.4.1 message

protobuf中定义一个消息类型是通过关键字message字段指定的，这个关键字类似于C++/Java中的class关键字。使用protobuf编译器将proto编译成C++代码之后，每个message都会生成一个名字与之对应的C++类，该类公开继承自google::protobuf::Message。

声明protobuf版本

syntax="proto3";//protobuf版本

声明定义message或service所在包（c++中对应为namespace）

package fixbug;//声明代码所在包（namespace）
package a.fixbug;//二级

message实例

message有三种成员类型：数据、列表、映射表

message ResultCode{
    int32 errcode=1;
    bytes errmsg=2;
}
//登录消息类型
message LoginRequest{
    bytes name=1;//1:消息类型字段
    bytes pwd=2;
    // map<int32,string> test=3; //映射表
}

//登录相应消息类型
message LoginResponse{
    ResultCode result=1;//1为标识号
    bool success=3;
}

//定义User类型
message User{
    bytes name=1;
    uint32 age=2;
    enum Sex{//枚举
        MAN=0;
        WOMAN=1;
    }
    Sex sex=3;
}

message GetFriendListsResponse{
    ResultCode result=1;
    repeated User friend_list=2;//列表类型
}

message字段规则

  • required:消息体中必填字段，不设置会导致编解码异常。
  • optional: 消息体中可选字段，可通过default关键字设置默认值。
  • repeated: 消息体中可重复字段，重复的值的顺序会被保留。其中，proto3默认使用packed方式存储，这样编码方式比较节省内存。

在proto3版本中，字段规则上移除了required，并把optional字段改名为singular。所有没有指定字段规则的字段默认为optional，

标识号

在消息体的定义中，每个字段都必须要有一个唯一的标识号，标识号是[0,2^29-1]范围内的一个整数

数据定义

许多标准的简单数据类型都可以用作message字段类型，包括bool,int32,float,double和string。还可以使用其他message类型作为字段类型在消息体中添加更多结构，如上面的ResultCode。

函数方法

用message关键字声明的的消息体，允许你检查、操作、读、或写整个消息，包括解析二进制字符串，以及序列化二进制字符串。除此之外，也定义了下列方法：

LoginResponse:缺省的构造函数。

~LoginResponse():缺省的析构函数。

LoginResponse(const LoginResponse& other):拷贝构造函数。

LoginResponse& operator=(const LoginResponse& other):
赋值 (Assignment ）操作符。

const UnknownFieldSet& unknown_fields() const:
返回当解析信息时遇到的未知字段的集合。

UnknownFieldSet* mutable_unknown_fields():
返回当前解析信息时遇到的未知字段的集合的一个mutale指针。

message实例

 LoginRequest req;
 req.set_name("wang");
 req.set_pwd("123434");
 //对象序列化
 std::string send_str;
 if (req.SerializeToString(&send_str)) {
   std::cout << send_str << std::endl;
 }

 // send_str反序列化
 LoginRequest reqB;
 if (reqB.ParseFromString(send_str)) {
   std::cout << reqB.name() << std::endl;
   std::cout << reqB.pwd() << std::endl;
 }
//GetFriendListsResponse 定义列表成员示例
 GetFriendListsResponse rsp;
 ResultCode *rc = rsp.mutable_result();
 rc->set_errcode(0);

 User *user1 = rsp.add_friend_list();
 user1->set_name("zhang san");
 user1->set_age(10);
 user1->set_sex(User::MAN);//设置枚举

 std::cout << rsp.friend_list_size() << std::endl;

protobuf编译

g++ main.cc test.pb.cc -o t -std=c++11 -L /usr/local/lib -lprotobuf -pthread

1.4.2 service

message定义了消息类型，service则定义了方法

protobuf默认不生成service类，需使用如下option开启：

option cc_generic_services=true;

service实例

service UserServiceRpc{
    rpc Login(LoginRequest) returns(LoginResponse);
    rpc GetFriendLists(GetFriendListsQuest) returns(GetFriendListsResponse);
}

定义的UserServiceRpc默认会生成两个类：UserServiceRpc和UserServiceRpc_Stub

UserServiceRpc

是被调用类，其继承自protobuf的service基类

class UserServiceRpc : public ::PROTOBUF_NAMESPACE_ID::Service {
 protected:
  // This class should be treated as an abstract interface.
  inline UserServiceRpc() {};
 public:
  virtual ~UserServiceRpc();

  typedef UserServiceRpc_Stub Stub;

  static const ::PROTOBUF_NAMESPACE_ID::ServiceDescriptor* descriptor();

  virtual void Login(::PROTOBUF_NAMESPACE_ID::RpcController* controller,
                       const ::fixbug::LoginRequest* request,
                       ::fixbug::LoginResponse* response,
                       ::google::protobuf::Closure* done);
  virtual void GetFriendLists(::PROTOBUF_NAMESPACE_ID::RpcController* controller,
                       const ::fixbug::GetFriendListsQuest* request,
                       ::fixbug::GetFriendListsResponse* response,
                       ::google::protobuf::Closure* done);

  // implements Service ----------------------------------------------

  const ::PROTOBUF_NAMESPACE_ID::ServiceDescriptor* GetDescriptor();
}

UserServiceRpc_Stub

UserServiceRpc_Stub是调用类，其继承自UserServiceRpc类，通过使用此类来调用方法，

注意，其没有默认的初始化构造函数，需通过RpcChannel指针来初始化

class UserServiceRpc_Stub : public UserServiceRpc {
 public:
  UserServiceRpc_Stub(::PROTOBUF_NAMESPACE_ID::RpcChannel* channel);
  UserServiceRpc_Stub(::PROTOBUF_NAMESPACE_ID::RpcChannel* channel,
                   ::PROTOBUF_NAMESPACE_ID::Service::ChannelOwnership ownership);
  ~UserServiceRpc_Stub();
  void Login(::PROTOBUF_NAMESPACE_ID::RpcController* controller,
                       const ::fixbug::LoginRequest* request,
                       ::fixbug::LoginResponse* response,
                       ::google::protobuf::Closure* done);
  void GetFriendLists(::PROTOBUF_NAMESPACE_ID::RpcController* controller,
                       const ::fixbug::GetFriendListsQuest* request,
                       ::fixbug::GetFriendListsResponse* response,
                       ::google::protobuf::Closure* done);
 private:
  ::PROTOBUF_NAMESPACE_ID::RpcChannel* channel_;
  bool owns_channel_;
}

注意看Login和GetFriendLists方法实现如下

void UserServiceRpc_Stub::Login(::PROTOBUF_NAMESPACE_ID::RpcController* controller,
                              const ::fixbug::LoginRequest* request,
                              ::fixbug::LoginResponse* response,
                              ::google::protobuf::Closure* done) {
  channel_->CallMethod(descriptor()->method(0),
                       controller, request, response, done);
}
void UserServiceRpc_Stub::GetFriendLists(::PROTOBUF_NAMESPACE_ID::RpcController* controller,
                              const ::fixbug::GetFriendListsQuest* request,
                              ::fixbug::GetFriendListsResponse* response,
                              ::google::protobuf::Closure* done) {
  channel_->CallMethod(descriptor()->method(1),
                       controller, request, response, done);
}

两个方法都是通过调用channel的CallMethod来调用，因此需要使用一个MyRpcChannel类继承RpcChannel类，实现其CallMethod方法来完成序列号、调用等操作。

2. mprpc实现

如上图，黄色部分的序列化和反序列化都是通过调用protobuf来完成，而mprpc框架主要使用muduo来接受请求调用然后调用相应函数，最后返回结果。

Caller的功能由RpcChannel类来实现，而Callee的功能由RpcProvidre类来实现。

2.1 RpcProvidre类

RpcProvider类需要实现接受请求，然后将其解析得到需要调用的方法和参数，然后调用相应方法和参数，最后将调用结果序列化发送给调用方。

私有成员

将OnMessage绑定到muduo库，当有消息到来时，所有反序列化、调用函数等操作都在OnMessage函数中进行

最后将调用结果发送到调用方是在SendRpcResponse方法中进行

//组合eventloop
muduo::EventLoop m_eventLoop;

// service服务类型信息
struct ServiceInfo {
  google::protobuf::Service *m_service;
  std::unordered_map<std::string, const google::protobuf::MethodDescriptor *>
      m_methodMap;  //保存服务方法
};

//存储服务对象和其服务方法的所有信息
std::unordered_map<std::string, ServiceInfo> m_serviceMap;

void OnConnection(const muduo::TcpConnectionPtr &);
void OnMessage(const muduo::TcpConnectionPtr &, muduo::Buffer *,
               muduo::Timestamp);
// closure回调操作，用于序列号rpc的响应和网络发送
void SendRpcResponse(const muduo::TcpConnectionPtr &,
                     google::protobuf::Message *);

公有成员

用户通过NotiFyService发布方法，然后调用Run函数即可

//发布Rpc方法的函数 主要是通过遍历将service中的函数指针和函数名都保存起来，方便调用
void NotiFyService(google::protobuf::Service *service);
//启动rpc服务节点，开始提供rpc远程网络调用服务：绑定消息函数，启动server监听
void Run();

关键函数

OnMessage（当请求到来，如何进行处理）：

  • 通过请求头大小及请求参数大小等解析出请求service_name和method_name及请求参数，然后根据服务名和方法名在存储服务及对应方法的map中找到对应的类指针，然后就通过该指针调用相应方法。
  • 注意看以下代码，调用方法是使用service的CallMethod方法进行（每个service都会实现一个CallMethod方法，传入对应方法就可以调用），而CallMethod最后一个参数是回调函数，当CallMethod执行完后会回调done-run()方法。在这里使用SendRpcResponse定义了一个closure类用于回调。
//获取service对象
google::protobuf::Service *service = it->second.m_service;
//获取method方法
const google::protobuf::MethodDescriptor *method = mit->second;

//生成rpc方法调用的请求request和响应response参数
google::protobuf::Message *request =
    service->GetRequestPrototype(method).New();
if (!request->ParseFromString(args_str)) {
  LOG_INFO("request parse error! content:%s", args_str.c_str());
}
google::protobuf::Message *response =
    service->GetResponsePrototype(method).New();

// method方法绑定一个closure的回调函数
//相当于使用SendRpcResponse函数重写了closure类中的Run方法,在用户自己写得服务中最后会执行done->run(),最终会执行SendRpcResponse方法
google::protobuf::Closure *done =
    google::protobuf::NewCallback<RpcProvider,
                                  const muduo::TcpConnectionPtr &,
                                  google::protobuf::Message *>(
        this, &RpcProvider::SendRpcResponse, conn, response);

//调用rpc方法
service->CallMethod(method, nullptr, request, response, done);

2.2 MprpcChannel类

在使用中，用户都是通过调用protobuf提供的ServiceRpc_Stub类来调用函数的，而ServiceRpc_Stub初始化需要一个RpcChannel，当调用ServiceRpc_Stub的函数时最后都会转到调用RpcChannel的CallMethod方法，因此需要将MprpcChannel类继承rpcchannel类，实现其虚有方法CallMethod即可。

例如，自定义的一个方法，使用stub调用时其底层是使用CallMethod进行调用：

void FriendServiceRpc_Stub::getFriendList(::PROTOBUF_NAMESPACE_ID::RpcController* controller,
                              const ::fixbug::getFriendListRequest* request,
                              ::fixbug::getFriendListResponse* response,
                              ::google::protobuf::Closure* done) {
  channel_->CallMethod(descriptor()->method(0),
                       controller, request, response, done);
}

MprpcChannel类只有一个成员：

//所有通过stub代理对象调用的rpc方法，都到这里做rpc方法调用数据的序列化和网络发送
void CallMethod(const google::protobuf::MethodDescriptor* method,
                google::protobuf::RpcController* controller,
                const google::protobuf::Message* request,
                google::protobuf::Message* response,
                google::protobuf::Closure* done) override;

在CallMethod方法中，主要是将要调用的方法名和参数进行序列化然后发送到远程，然后等待其返回结果，最后将返回结果反序列化即可。

2.3 mprpccontroller类

此类主要是用来控制rpc调用过程中出现的错误信息，如设置错误，获取错误信息。

2.4 Mprpc框架使用

先使用protobuf定义好远程调用service、方法和参数，比如定义提供FriendServiceRpc，其中包括GetFriendList方法

callee

class FriendService : public fixbug::FriendServiceRpc {
 public:
  std::vector<std::string> GetFriendList(uint32_t userid);

  void getFriendList(google::protobuf::RpcController* controller,
                     const ::fixbug::getFriendListRequest* request,
                     ::fixbug::getFriendListResponse* response,
                     ::google::protobuf::Closure* done) ;
};

对于提供服务者，定义FriendService类继承FriendServiceRpc类，实现其虚有方法getFriendList，为使逻辑清晰，注意在此getFriendList方法中只是将设计的远程调用参数转为本地调用需要的参数，然后调用业务逻辑，最后将返回的参数打包成远程接口参数，并没有在此函数中执行真正的业务逻辑。

最后，用户只需要将此业务注册发布即可

int main(int argc, char** argv) {
  LOG_INFO("first log message test");
  LOG_ERROR("%s:%s:%d",__FILE__,__FUNCTION__,__LINE__);
  //框架初始化
  MprpcApplication::Init(argc, argv);

  RpcProvider provider;
  provider.NotiFyService(new FriendService());

  provider.Run();

  return 0;
}

caller

调用方都是通过protobu生成的service_stub去调用，只需要在定义service_stub变量时使用mprpcchannel初始化，然后定义请求参数和接受参数变量，再调用stub的方法即可。

int main(int argc, char **argv) {
  //需要先调用框架的初始化函数
  MprpcApplication::Init(argc, argv);

  //调用远程发布的rpc方法Login
  fixbug::FriendServiceRpc_Stub stub(new MprpcChannel());
  //相当于调用RpcChannel->RpcChannel::callMethod
  // stub.Login();
  fixbug::getFriendListRequest request;
  request.set_id(1);

  fixbug::getFriendListResponse response;
  MprpcController controller;

  stub.getFriendList(&controller, &request, &response, nullptr);

  if (controller.Failed()) {
    LOG_INFO("%s", controller.ErrorText().c_str());
  } else {
    if (response.result().errcode() == 0) {
      LOG_INFO("rpc login response: ");
      for (auto name : response.friends()) {
        LOG_INFO("%s", name.c_str());
      }
    } else {
      LOG_INFO("rpc login response error:%s",
               response.result().errmsg().c_str());
    }
  }

  return 0;
}
